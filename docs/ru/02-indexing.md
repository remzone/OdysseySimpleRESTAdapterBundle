# Индексирование

## Обзор

Все API-ответы отдаются из индекса, а не напрямую из базы данных Pimcore.

Это одна из ключевых причин использовать bundle:

- более предсказуемое время ответа
- меньшая нагрузка на базу данных
- лучшая пригодность для поисковых и интеграционных read-heavy сценариев

## Search backend

Текущий fork Odyssey поддерживает и OpenSearch, и Elasticsearch через конфигурируемую фабрику клиента.

Практически это означает:

- используйте `simple_rest_adapter.engine` для выбора backend
- используйте `simple_rest_adapter.hosts` для списка хостов
- исторические имена вроде `datahub_es_index_queue` сохранены для обратной совместимости
- `es_hosts` по-прежнему работает как fallback-настройка

## Модель создания индексов

Каждая конфигурация DataHub создаёт собственный набор индексов.

Обычно это отдельные индексы для:

- классов DataObject
- папок DataObject
- Assets
- папок Assets

## Очередь и асинхронная обработка

Индексирование работает асинхронно через Symfony Messenger.

Настроенный transport:

```text
datahub_es_index_queue
```

Типовая команда воркера:

```bash
php bin/console messenger:consume datahub_es_index_queue --memory-limit=250M --time-limit=3600
```

## Что запускает обновление индекса

Создание и обновление индексов происходит, когда:

- создаются или меняются endpoint configurations
- создаются, меняются или удаляются индексируемые элементы
- воркер обрабатывает сообщения из очереди

Этот fork также диспатчит события жизненного цикла конфигурации:

- `datahub.simple_rest.configuration.pre_save`
- `datahub.simple_rest.configuration.post_save`

Это полезно, если вы расширяете bundle и хотите реагировать на изменения конфигурации.

## Дерево и виртуальные папки

Индексное дерево не всегда является прямой копией дерева папок Pimcore.

Bundle может:

- переписывать пути относительно границ workspace
- вычислять общий родительский путь
- создавать виртуальные папки, чтобы сохранить навигационную иерархию при пропуске промежуточных узлов

## Замечание по tokenizer

Если ваш search backend ожидает стиль Elasticsearch 8, используйте:

```yaml
type: ngram
```

вместо:

```yaml
type: nGram
```

## Практические рекомендации

Для production-сценариев:

- держите messenger worker постоянно запущенным
- следите за ростом очереди
- выполняйте переиндексацию после крупных изменений схемы
- проверяйте появление индексов в OpenSearch после обновления endpoint'ов
